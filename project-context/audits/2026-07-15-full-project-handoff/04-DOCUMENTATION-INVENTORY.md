## Documentation gaps

### Package-level gaps

- `packages/cli/` has no package README covering installation, binary name, commands, exit behavior, offline registry packaging, version compatibility, or removal.
- `packages/core/` has no README or public API reference for registry/manifest exports and validation behavior.
- `packages/harness/` has no README explaining acceptance-harness scope, intended consumers, fixtures, and stability.
- `packages/ui/README.md` provides boundaries and a usage snippet but is not a complete component/API reference.

### Registry-item contract gaps

- None of the three component Markdown files contains explicit **removal/uninstall** instructions.
- None of the three component Markdown files contains an explicit **compatibility** section.
- None of the three component Markdown files names a **test command** or explains how consumers should verify the copied source after customization.
- The pattern and example notes do not document security boundaries, accessibility implications, tests, compatibility, or removal.
- The registry manifests carry machine-readable dependencies and test metadata, but the human-readable docs do not consistently expose those facts.

### Public-guide gaps

- There is no consolidated API reference for `@agents-sdk/core`, `@agents-sdk/ui`, `@agents-sdk/cli`, or `@agents-sdk/harness`.
- There is no dedicated authentication/provider integration guide, transport/streaming guide, error-handling guide, accessibility guide, testing guide, upgrade guide, or uninstall/removal guide.
- Agents & Skills and Tools & MCP have no proven installable primitives or documentation; repository policy correctly forbids empty capability pages.
- The two example Markdown files are only three lines each and do not explain architecture, expected behavior, test coverage, extension points, or failure modes.
- There is no single docs index mapping repository documentation, registry item docs, schemas, public TSX pages, and package references.
- Public docs copy is embedded in page components, with no documented content-authoring, navigation, search, versioning, or deprecation workflow.

### Context-maintenance gaps

- The divergent `* 2.*` files make search results ambiguous and can cause an agent to load superseded Agenty facts.
- `project-context/tasks.md` and `project-context/tasks.json` are paired human/machine views, but no checked synchronization rule is evident from the audited documentation.
- Root `DESIGN.md` and `project-context/DESIGN.md` have adjacent names and scopes; their ownership distinction should remain explicit to avoid drift.
- Root release/status statements, `CHANGELOG.md`, `project-context/CURRENT_STATE.md`, and `project-context/plan.md` overlap and require a documented update order.

## Recommended documentation roadmap

### Phase 0 — Resolve source-of-truth ambiguity

1. Review all twenty `* 2.*` files for historically unique decisions, prompts, verification evidence, and task provenance.
2. Move valuable unique material into a dated migration archive or the appropriate canonical ledger.
3. Remove the collision copies from the active repository tree after reconciliation.
4. Add a short source-of-truth map linking root governance, project context, registry docs, templates, schemas, package docs, and public routes.

### Phase 1 — Complete v0.1 item documentation

1. Define a required item-doc template: purpose, installation, files written, API/anatomy, dependencies, compatibility, security boundary, accessibility, states/failures, tests, customization, upgrade behavior, and removal.
2. Bring the three component docs, one pattern doc, two example docs, and template README up to that contract.
3. Add a validation check that compares documented claims and required sections with registry manifest metadata.
4. Keep manifest and documentation changes atomic so item versions remain truthful.

### Phase 2 — Add package-consumer references

1. Create package READMEs for CLI, core, and harness.
2. Expand the UI README into an export/component reference while keeping host-specific adapters outside the package boundary.
3. Document supported React, Node, package-manager, module-format, and TypeScript versions in one canonical compatibility matrix.
4. Document install, update, diff, doctor, rollback/conflict behavior, and removal end to end.

### Phase 3 — Establish a maintainable docs system

1. Create a canonical docs index and information architecture covering Getting Started, Concepts, Components, Patterns, Templates, Examples, CLI, API, Security, Accessibility, Testing, and Contribution.
2. Decide whether public prose remains TSX-owned or moves to Markdown/MDX; document the authoring boundary either way.
3. Add link checking, docs build validation, registry-doc completeness checks, and stale/generated mirror checks to `pnpm validate`.
4. Define versioning, deprecation, migration, and release-note linkage for public docs.

### Phase 4 — Expand only with runnable evidence

1. Publish Agents & Skills documentation only when at least one installable, tested, versioned item exists.
2. Publish Tools & MCP documentation only when an integration has explicit auth/scopes, security boundaries, tests, and removal steps.
3. Keep every new public category evidence-gated according to `AGENTS.md`, `project-context/ACCEPTANCE.md`, and the registry contract.
